Panorama das Etapas — Nível 1

Do que você já tem (ETL em pandas rodando local) até o ciclo completo na nuvem: extração → Bronze/Silver/Gold no Supabase → agendamento no GitHub Actions → dashboard em Streamlit. Sete etapas, na ordem em que você deve executá-las.

Excel→ Python/pandas→ Supabase (Bronze·Silver·Gold)→ GitHub Actions (cron)→ Streamlit

Como vamos trabalhar. Sou seu professor nesse projeto. A regra do Nível 1 é uma só: fechar o ciclo ponta a ponta o mais rápido possível, mesmo que tosco. Nada de otimizar antes de estar rodando. Fazemos uma etapa, provamos que funciona (o "critério de pronto" de cada card), e só então avançamos.

Você já tem o mais difícil: um ETL que separa Bronze/Silver/Gold e carrega no Postgres. As etapas 1–3 são para rodar isso de forma confiável na sua máquina; 4–5 colocam na nuvem e agendam; 6–7 entregam o dashboard e revisam o ciclo.

As 7 etapas do Nível 1

1

Rodar o ETL local sem erro

Provar que etl_estoque.py extrai, transforma e gera Bronze/Silver/Gold.

✅ Concluído

Antes de qualquer nuvem, o pipeline precisa rodar limpo na sua máquina. Aqui você monta o ambiente e corrige o que impede a execução.

O que fazer
  • Criar um ambiente virtual (python -m venv .venv) e instalar dependências: pandas, openpyxl, sqlalchemy, psycopg2-binary, python-dotenv.
  • Congelar em requirements.txt (pip freeze > requirements.txt) — o GitHub Actions vai precisar disso na Etapa 5.
  • Corrigir o caminho dos dados. Hoje o script usa os.path.join(dirname, "..", "data"), que sobe uma pasta. Como o script está na raiz junto de data/, isso aponta para o lugar errado. Ajuste para os.path.join(dirname, "data") (ou mova o script para uma subpasta src/).
  • Nesta etapa, teste só extract() e transform() — comente a chamada a load() e dê print nos DataFrames.
⚠️ Ainda não conecte no banco. Primeiro garanta que os dados saem corretos do transform(): confira as datas, a padronização de loja (L0…) e o Gold agregado por loja+categoria.
Critério de pronto: print mostra as 3 tabelas (bronze, silver, gold) com linhas coerentes e sem exceção.
O que aconteceu na prática
  • Ambiente ativado no PowerShell com .venv\Scripts\Activate.ps1 (prompt do Antigravity é PowerShell, não bash/cmd).
  • Caminho corrigido para os.path.join(os.path.dirname(__file__), "data"), conforme previsto.
  • load() comentada em run() e adicionados print(bronze.head()), print(silver.head()), print(gold.head()) antes do print final de OK.
  • Bug real encontrado na 1ª rodada: pd.to_datetime(s["data"], format="mixed", dayfirst=True) trocava dia/mês também nas datas já em ISO (2026-06-01 virava 6 de janeiro em vez de 1º de junho). Corrigido separando as linhas por formato (contém - = ISO, parseado sem dayfirst; senão = BR, parseado com dayfirst=True) e recombinando com pd.concat(...).sort_index().
  • Segundo bug: a regra "LOJA " → "L0" só cobria o formato por extenso; códigos já abreviados como L02 não batiam com L002/L01/L001 e viravam lojas "diferentes" no Gold. Corrigido extraindo os dígitos por regex (str.extract(r"(\d+)")) e remontando sempre com 3 dígitos ("L" + numero.zfill(3)).
  • Importante: os Excel de origem em data/ permanecem sujos de propósito (não foram editados) — a limpeza acontece inteiramente dentro de transform(); o Bronze preserva o dado bruto tal como veio.
2

Criar o projeto no Supabase

Seu Postgres gerenciado na nuvem — de graça para começar.

✅ Concluído

O Supabase é um Postgres hospedado. É onde Bronze, Silver e Gold vão viver. Você só precisa do banco — ignore Auth, Storage e o resto por enquanto.

O que fazer
  • Criar conta e um novo projeto; guardar a senha do banco.
  • Pegar a connection string (Project Settings → Database → Connection string, formato URI). É a variável SUPABASE_URL que o script já espera em os.environ["SUPABASE_URL"].
  • Testar a conexão localmente com um script curto ou psql.
⚠️ A connection string tem senha. Ela vai num .env que nunca entra no Git (Etapa 4). Nunca cole isso direto no código.
Critério de pronto: você consegue conectar ao banco de fora e listar (ainda vazio) sem erro de autenticação/SSL.
O que aconteceu na prática
  • Projeto criado no Supabase e SUPABASE_URL salva em .env na raiz do projeto (mesma pasta do etl_estoque.py), formato postgresql://postgres:senha@db.xxxx.supabase.co:5432/postgres.
  • Conexão testada sem script separado — um one-liner direto no terminal:
    python -c "from dotenv import load_dotenv; import os; from sqlalchemy import create_engine, text; load_dotenv(); engine = create_engine(os.environ['SUPABASE_URL']); conn = engine.connect(); print(conn.execute(text('SELECT version()')).fetchone()); conn.close()"
  • Retornou PostgreSQL 17.6 on x86_64-pc-linux-gnu... — conexão confirmada.
3

Criar os schemas Bronze / Silver / Gold e carregar

A arquitetura Medalhão dentro do Postgres.

✅ Concluído

O load() escreve em dw_bronze, dw_silver e dw_gold. Esses schemas precisam existir, e a tabela dw_gold.posicao_estoque precisa existir (o código faz truncate nela antes do append).

O que fazer
  • No SQL Editor do Supabase: create schema dw_bronze; create schema dw_silver; create schema dw_gold;
  • Criar a tabela dw_gold.posicao_estoque com as colunas do Gold (cod_loja, categoria, qtd_entradas, qtd_saidas, saldo, valor_saldo).
  • Reativar o load() e rodar o run() completo.
  • Conferir no Supabase: as 3 tabelas populadas.
● Bronze — cópia fiel do bruto ● Silver — limpo e enriquecido ● Gold — KPI por loja+categoria
Critério de pronto: python etl_estoque.py imprime OK - bronze=… silver=… gold=… e as tabelas aparecem no Supabase.
O que aconteceu na prática
  • Schemas criados com create schema if not exists dw_bronze/dw_silver/dw_gold;.
  • Tabela dw_gold.posicao_estoque criada manualmente com create table if not exists e colunas cod_loja text, categoria text, qtd_entradas numeric, qtd_saidas numeric, saldo numeric, valor_saldo numeric.
  • Os prints de debug da Etapa 1 (bronze.head(), silver.head(), gold.head()) foram removidos e a chamada a load() reativada em run().
  • python etl_estoque.py imprimiu OK - bronze=... silver=... gold=... e as 3 tabelas apareceram populadas no Table Editor do Supabase.
4

Versionar no GitHub (com segredos fora)

Repo limpo, .gitignore e .env.example.

✅ Concluído

Para agendar na nuvem, o código precisa estar no GitHub. O ponto crítico aqui é não vazar a connection string.

O que fazer
  • git init, criar .gitignore incluindo .venv/, .env, __pycache__/.
  • Criar um .env.example (sem a senha real) documentando SUPABASE_URL=.
  • Commit e push para um repositório novo no GitHub.
  • Conferir no GitHub que o .env não subiu.
O que aconteceu na prática
  • Repositório criado antes pelo navegador (github.com/new), sem README/gitignore/license iniciais, para evitar conflito no primeiro push.
  • git status revelou que .claude/settings.local.json (config local do Claude Code, sem segredos) também seria commitado — adicionado ao .gitignore e removido do stage com git rm --cached antes do commit.
  • git commit falhou na primeira tentativa por falta de identidade Git — resolvido com git config --global user.name e user.email.
  • git remote add origin ... + git branch -M main + git push -u origin main — autenticação via navegador, push concluído.
  • Confirmado visualmente no GitHub: .env ausente, só .env.example presente.
Critério de pronto: repo online, com etl_estoque.py, requirements.txt, data/ e .env.example — e sem nenhum segredo.
5

Agendar com GitHub Actions (cron)

O "orquestrador" do Nível 1 — sem Airflow, sem Dagster.

✅ Concluído

Aqui está o coração do aprendizado de deploy deste nível: um workflow que instala as deps e roda o ETL num horário fixo, na infraestrutura do GitHub.

O que fazer
  • Criar .github/workflows/etl.yml com gatilho schedule: cron + workflow_dispatch (para rodar manualmente no teste).
  • Steps: checkout → setup-python → pip install -r requirements.txt → python etl_estoque.py.
  • Guardar a connection string em Settings → Secrets and variables → Actions como SUPABASE_URL, e injetar via env: no step.
  • Disparar manualmente pela aba Actions e ler os logs.
⚠️ Lembrete: cron do GitHub Actions roda em UTC. Ajuste o horário considerando seu fuso (-03).
O que aconteceu na prática
  • Secret SUPABASE_URL cadastrado em Settings → Secrets and variables → Actions.
  • 1ª tentativa (erro): sqlalchemy.exc.ArgumentError: Could not parse SQLAlchemy URL — o valor colado no secret estava malformado (provável prefixo/aspas/espaço extra). Corrigido recadastrando só a URI pura.
  • 2ª tentativa (erro): psycopg2.OperationalError: Network is unreachable ao conectar em endereço IPv6 — runners do GitHub Actions não têm saída IPv6, e a conexão direta do Supabase (db.xxx.supabase.co:5432) resolve para IPv6. Funcionava local só porque a rede local tem IPv6.
  • Correção: trocado o connect method no Supabase (Project Settings → Database → Connection string) de "Direct connection" para "Session pooler" (compatível com IPv4, host aws-0-<região>.pooler.supabase.com). String atualizada tanto no .env local quanto no secret do GitHub, para manter os dois ambientes consistentes.
  • 3ª tentativa: execução manual via Run workflow ficou verde.
Critério de pronto: execução manual verde na aba Actions e os dados atualizados no Supabase.
6

Dashboard em Streamlit sobre o Gold

O consumo final — Python puro, sem front-end.

~2h

O Streamlit lê apenas o schema Gold (nunca o Silver/Bronze) e mostra os KPIs. Regra de ouro da Medalhão: dashboard consome Gold.

O que fazer
  • Criar app.py que conecta no Supabase e faz SELECT * FROM dw_gold.posicao_estoque.
  • Exibir: tabela, saldo por loja, valor de saldo por categoria (um ou dois gráficos st.bar_chart).
  • Rodar local com streamlit run app.py.
  • Publicar no Streamlit Community Cloud (grátis), apontando para o repo e configurando o secret SUPABASE_URL.
Critério de pronto: URL pública do Streamlit mostrando os KPIs vindos do Gold.
7

Fechar e revisar o ciclo

Provar que tudo se conecta e documentar.

~1h

O Nível 1 termina quando você consegue disparar o pipeline e ver o dashboard refletir a mudança — sem tocar em nada manualmente entre as pontas.

O que fazer
  • Alterar/adicionar uma linha nos Excel de origem, rodar o Actions (ou esperar o cron) e confirmar que o Streamlit muda.
  • Escrever um README.md curto: como rodar local, como o cron funciona, onde estão os segredos.
  • Anotar limitações (ex.: Excel ainda é a fonte, sem testes de dados) — isso vira a lista de entrada do Nível 2.
Critério de pronto (Nível 1 concluído): Excel → Actions → Supabase → Streamlit funcionando de ponta a ponta, na nuvem, com um README.

O que NÃO fazer no Nível 1

Não introduza dbt, Dagster, Airflow, Docker ou API de ERP agora — tudo isso é Nível 2/3. Adicionar peças cedo é o erro clássico que trava o aprendizado de deploy.

Não tente deixar o ETL "perfeito". Bronze como cópia fiel e Silver com limpeza simples bastam. O objetivo é o ciclo, não a transformação.

Quando as 7 etapas estiverem verdes, me chame: montamos juntos o passo a passo detalhado da Etapa 1 (ambiente + correção do caminho) e seguimos card a card.
