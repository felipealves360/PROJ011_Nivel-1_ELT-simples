1

ETL rodando local — ainda sem banco

Provar que os dados saem corretos do transform() antes de qualquer nuvem.

~40 min

Esta é a etapa mais importante e a mais fácil de subestimar. A regra: não conecte no banco enquanto não olhar os dados com os próprios olhos. Se você carregar dado sujo, vai debugar no lugar errado depois.

Como o script se organiza
  • extract() — lê os dois Excel, devolve dois DataFrames crus.
  • transform() — devolve três DataFrames: bronze (cópia fiel), silver (limpo), gold (agregado).
  • load() — escreve os três no Postgres. Nesta etapa fica comentado.
etl_estoque.py — extract 
import os
import pandas as pd
from sqlalchemy import create_engine, text
from dotenv import load_dotenv

load_dotenv()
DATA = os.path.join(os.path.dirname(__file__), "data")  # pasta data/ ao lado do script


def extract():
    mov  = pd.read_excel(f"{DATA}/2026-06-26_dados-brutos_movimentacao-estoque.xlsx")
    prod = pd.read_excel(f"{DATA}/2026-06-26_dados-brutos_dim-produtos.xlsx")
    return mov, prod
🐞 Armadilha real: se o script está na raiz, junto da pasta data/, o caminho é os.path.join(dirname, "data"). Um ".." a mais (join(dirname, "..", "data")) sobe uma pasta e aponta pro lugar errado. Errar isso dá FileNotFoundError logo de cara.
O transform — cada linha resolve um problema de dado sujo

Os Excel de origem são sujos de propósito. Nunca "arrume o Excel": toda limpeza mora aqui, e o Bronze guarda o bruto exatamente como veio.

etl_estoque.py — transform 
def transform(mov, prod):
    # ---- BRONZE: copia fiel do bruto (so para inspecao) ----
    bronze = mov.copy()

    # ---- SILVER: limpeza observavel ----
    s = mov.copy()
    for col in ["cod_loja", "tipo_movimentacao", "sku"]:
        s[col] = s[col].astype(str).str.strip().str.upper()      # 1) espacos + caixa

    s["cod_loja"] = s["cod_loja"].str.replace("LOJA ", "L", regex=False)
    num_loja = s["cod_loja"].str.extract(r"(\d+)")[0].astype(int)
    s["cod_loja"] = "L" + num_loja.astype(str).str.zfill(3)     # 2) padroniza loja

    iso      = s["data"].str.contains("-", na=False)
    data_iso = pd.to_datetime(s.loc[iso,  "data"], format="%Y-%m-%d")
    data_br  = pd.to_datetime(s.loc[~iso, "data"], dayfirst=True)
    s["data"] = pd.concat([data_iso, data_br]).sort_index()      # 3) datas mistas

    neg = (s["tipo_movimentacao"] == "ENTRADA") & (s["quantidade"] < 0)
    s.loc[neg, "quantidade"] = s.loc[neg, "quantidade"].abs()      # 4) regra de negocio
    s = s.dropna(subset=["quantidade", "valor_unitario"])          # 5) nulos criticos
    s = s.drop_duplicates(
        subset=["data", "cod_loja", "sku", "tipo_movimentacao", "quantidade"]
    )                                                             # 6) duplicatas

    # dimensao de produtos: limpa e deduplica pela chave estavel (sku)
    p = prod.copy()
    p["sku"] = p["sku"].astype(str).str.strip().str.upper()
    p["categoria"] = p["categoria"].fillna("SEM CATEGORIA").astype(str).str.strip().str.upper()
    p = p.drop_duplicates(subset=["sku"], keep="first")

    silver = s.merge(p[["sku", "categoria"]], on="sku", how="left")
    silver["categoria"] = silver["categoria"].fillna("SEM CATEGORIA")  # sku sem cadastro

    # ---- GOLD: agrega o KPI por loja + categoria ----
    silver["ent"] = silver["quantidade"].where(silver["tipo_movimentacao"] == "ENTRADA", 0)
    silver["sai"] = silver["quantidade"].where(silver["tipo_movimentacao"] == "SAIDA",   0)

    gold = silver.groupby(["cod_loja", "categoria"], as_index=False).agg(
        qtd_entradas=("ent", "sum"),
        qtd_saidas=("sai", "sum"),
        preco_medio=("valor_unitario", "mean"),
    )
    gold["saldo"]       = gold["qtd_entradas"] - gold["qtd_saidas"]
    gold["valor_saldo"] = (gold["saldo"] * gold["preco_medio"]).round(2)
    gold = gold[["cod_loja", "categoria", "qtd_entradas", "qtd_saidas", "saldo", "valor_saldo"]]

    silver = silver.drop(columns=["ent", "sai"])  # remove colunas auxiliares
    return bronze, silver, gold
🐞 Os dois bugs que custaram tempo (leia com atenção — eles voltam):
  • Datas mistas: a tentação é pd.to_datetime(s["data"], format="mixed", dayfirst=True). Isso corrompe as datas que já estão em ISO: 2026-06-01 vira 6 de janeiro. A solução é separar por formato (tem - = ISO, parseia sem dayfirst; senão = BR, com dayfirst=True) e recombinar com pd.concat(...).sort_index() — o sort_index() é o que devolve as linhas à ordem original.
  • Código de loja: só trocar "LOJA " → "L" não basta. O arquivo tem LOJA 2, L02 e L002 — sem normalizar os dígitos, viram três lojas diferentes no groupby do Gold. Por isso o str.extract(r"(\d+)") + zfill(3): extrai só o número e remonta sempre com 3 dígitos.
Rodar e conferir com os próprios olhos

    run()

Supabase: criar o banco e testar a conexão

Postgres gerenciado, grátis. Só o banco importa — ignore Auth/Storage.

~15 min
1. Criar o projeto
  • Criar conta em supabase.com → New project.
  • Guarde a senha do banco na hora (ela só é mostrada uma vez; depois só resetando).
2. Pegar a connection string — MODO SESSION POOLER

Project Settings → Database → Connection string → no seletor "Connect method", escolha Session pooler (NÃO "Direct connection"). Formato:

postgresql://postgres.<ref-do-projeto>:<SENHA>@aws-1-<regiao>.pooler.supabase.com:5432/postgres
🐞 A armadilha mais cara do projeto inteiro. A "Direct connection" (@db.xxxx.supabase.co) resolve para um endereço IPv6. Funciona na sua máquina (sua rede tem IPv6) e falha em toda nuvem — GitHub Actions e Streamlit Cloud não têm saída IPv6 → psycopg2.OperationalError: Network is unreachable. O Session pooler é IPv4. Adote ele desde o dia 1, nos três ambientes (local, Actions, Streamlit), pra nunca ter strings divergentes.
3. Criar o .env na raiz
.env (NUNCA vai pro Git) 
SUPABASE_URL=postgresql://postgres.abcdefgh:MinhaSenha@aws-1-sa-east-1.pooler.supabase.com:5432/postgres

Sem aspas, sem espaços, sem ponto-e-vírgula. O load_dotenv() lê esse arquivo e o script acessa via os.environ["SUPABASE_URL"].

4. Testar a conexão ANTES de escrever qualquer load()

Este one-liner é o melhor investimento do projeto: se ele retorna a versão do Postgres, rede e credencial estão OK, e qualquer erro futuro é seu código, não infra.

python -c "from dotenv import load_dotenv; import os; from sqlalchemy import create_engine, text; load_dotenv(); engine = create_engine(os.environ['SUPABASE_URL']); conn = engine.connect(); print(conn.execute(text('SELECT version()')).fetchone()); conn.close()"

Esperado: ('PostgreSQL 17.x on x86_64-pc-linux-gnu...',)

Critério de pronto: o one-liner imprime a versão do Postgres.

Schemas Bronze/Silver/Gold + carga real

A arquitetura Medalhão dentro do Postgres.

~20 min
1. Criar os schemas (SQL Editor do Supabase)
create schema if not exists dw_bronze;
create schema if not exists dw_silver;
create schema if not exists dw_gold;
2. Criar SÓ a tabela do Gold
create table if not exists dw_gold.posicao_estoque (
    cod_loja      text,
    categoria     text,
    qtd_entradas  numeric,
    qtd_saidas    numeric,
    saldo         numeric,
    valor_saldo   numeric
);
💡 Por que só o Gold? No load(), Bronze e Silver usam to_sql(if_exists="replace") — o pandas cria a tabela sozinho (e recria a cada rodada). O Gold usa truncate + append, para preservar os tipos que você definiu — e truncate numa tabela que não existe dá erro. Daí o CREATE TABLE manual.
3. O load()
etl_estoque.py — load 
def load(bronze, silver, gold):
    engine = create_engine(os.environ["SUPABASE_URL"])
    with engine.begin() as conn:            # begin() = tudo numa transacao (ou entra tudo, ou nada)
        bronze.to_sql("movimentacao", conn, schema="dw_bronze", if_exists="replace", index=False)
        silver.to_sql("movimentacao", conn, schema="dw_silver", if_exists="replace", index=False)
        conn.execute(text("truncate table dw_gold.posicao_estoque"))
        gold.to_sql("posicao_estoque", conn, schema="dw_gold", if_exists="append", index=False)
4. Reativar o load e rodar de verdade

Descomente load(bronze, silver, gold) no run(), remova os print de debug da Etapa 1, e rode:

python etl_estoque.py
# OK - bronze=... silver=... gold=...

Confirme no Table Editor do Supabase que as 3 tabelas apareceram populadas.

Critério de pronto: OK - bronze=… silver=… gold=… no terminal e as 3 tabelas visíveis no Supabase.

Agendamento com GitHub Actions

O "orquestrador" do Nível 1 — sem Airflow, sem Docker.

~30 min

Modelo mental: o GitHub liga uma VM Ubuntu vazia e efêmera, ela clona seu repo, instala Python, instala as deps, roda o script e é destruída. Nada persiste entre execuções — é por isso que requirements.txt precisa estar correto e o segredo precisa ser injetado toda vez.

1. Criar o workflow

O GitHub descobre automaticamente qualquer .yml dentro de .github/workflows/. O caminho é a única coisa que importa; o nome do arquivo é livre.

.github/workflows/etl.yml 
name: ETL Estoque  # nome exibido na aba Actions, so um rotulo

on:
  workflow_dispatch: {}   # habilita o botao "Run workflow" manual na aba Actions
  schedule:
    # cron roda sempre em UTC. "0 9 * * *" = todo dia as 09:00 UTC (06:00 no -03)
    - cron: "0 9 * * *"

jobs:
  run-etl:
    runs-on: ubuntu-latest   # VM efemera: nasce vazia, e destruida no fim
    steps:
      - uses: actions/checkout@v4       # clona o repo dentro da VM

      - uses: actions/setup-python@v5   # instala o Python na VM
        with:
          python-version: "3.12"

      - run: pip install -r requirements.txt   # instala as dependencias

      - run: python etl_estoque.py             # roda o ETL de verdade
        env:
          # injeta o secret so nesse processo; o GitHub mascara o valor nos logs
          SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
Ler o cron (5 campos)
┌─ minuto (0-59)
│ ┌─ hora (0-23)  ← em UTC!
│ │ ┌─ dia do mês (1-31)
│ │ │ ┌─ mês (1-12)
│ │ │ │ ┌─ dia da semana (0-6, 0=domingo)
│ │ │ │ │
0 9 * * *   →  todo dia, 09:00 UTC  =  06:00 em Brasília (-03)
⚠️ Sempre inclua workflow_dispatch. Sem ele você não consegue testar sem esperar o cron — e o cron do GitHub é "best effort", pode atrasar minutos.
2. Cadastrar o secret

Repo → Settings → Secrets and variables → Actions → New repository secret:

  • Name: SUPABASE_URL
  • Secret: só a URI pura — postgresql://postgres.xxx:senha@aws-1-....pooler.supabase.com:5432/postgres
🐞 Armadilha real: sqlalchemy.exc.ArgumentError: Could not parse SQLAlchemy URL. Causa: o valor colado no secret tinha lixo junto — o prefixo SUPABASE_URL=, aspas, ou um espaço/quebra de linha. O campo do secret é só o valor, não a linha do .env.
3. Rodar manualmente e ler o log

Aba Actions → workflow "ETL Estoque" → Run workflow → clique na execução e abra os steps.

🐞 Armadilha real: psycopg2.OperationalError: connection to server at "..." (2600:...), port 5432 failed: Network is unreachable. Aquele 2600:... é um IPv6. Runners do GitHub não têm saída IPv6. Solução: trocar a connection string para Session pooler no Supabase e atualizar nos dois lugares — o .env local e o secret do GitHub. (Se você já seguiu a Etapa 2 corretamente, isso nunca acontece.)
💡 Um warning de "Node.js 20 deprecation" no log é normal e não é seu problema — é o runtime interno das actions oficiais.
⚠️ Repositório sem nenhum push por ~60 dias: o GitHub pausa automaticamente os workflows agendados. O workflow_dispatch continua funcionando. Basta um push (ou reativar na aba Actions) pra religar o cron.
Critério de pronto: execução manual verde na aba Actions e os dados atualizados no Supabase.

Dashboard em Streamlit + deploy

Python puro, sem front-end. Lê só o Gold.

~40 min

Modelo mental do Streamlit: não existe "atualizar só um pedaço da tela". A cada interação (clique, filtro, F5) o script inteiro roda de novo, do topo ao fim. É por isso que a consulta ao banco precisa de cache — sem ele, cada clique abre uma conexão nova no Supabase.

1. Instalar e congelar
pip install streamlit
pip freeze | Out-File -Encoding utf8 requirements.txt   # regenera COM o streamlit dentro
⚠️ Se esquecer de regenerar o requirements.txt, o deploy no Streamlit Cloud falha (a VM dele instala só o que está no arquivo).
2. O app.py
app.py 
import os
import pandas as pd
import streamlit as st
from sqlalchemy import create_engine
from dotenv import load_dotenv

load_dotenv()  # le o .env local; no Streamlit Cloud quem injeta as env vars e o secret do painel

st.set_page_config(page_title="Posição de Estoque", layout="wide")  # 1a chamada st.*, obrigatorio


@st.cache_data(ttl=600)  # cacheia o resultado por 10min p/ nao bater no banco a cada interacao
def load_gold():
    engine = create_engine(os.environ["SUPABASE_URL"])
    # so le o schema Gold (regra do Medalhao: dashboard nunca toca Bronze/Silver)
    return pd.read_sql("select * from dw_gold.posicao_estoque", engine)


st.title("Posição de Estoque")
st.caption("Dados consumidos direto do schema dw_gold (Medalhão) no Supabase.")

df = load_gold()

col1, col2, col3 = st.columns(3)  # 3 colunas lado a lado, cada uma vira um KPI card
col1.metric("Saldo total (unid.)",     f"{df['saldo'].sum():,.0f}")
col2.metric("Valor total em estoque",  f"R$ {df['valor_saldo'].sum():,.2f}")
col3.metric("Lojas",                   df["cod_loja"].nunique())

st.subheader("Saldo por loja")
# as_index=True mantem cod_loja como indice -> vira o eixo X do grafico
saldo_loja = df.groupby("cod_loja", as_index=True)["saldo"].sum()
st.bar_chart(saldo_loja)

st.subheader("Valor em estoque por categoria")
valor_categoria = df.groupby("categoria", as_index=True)["valor_saldo"].sum()
st.bar_chart(valor_categoria)

st.subheader("Detalhe (loja x categoria)")
st.dataframe(df, width="stretch")  # tabela interativa (ordenavel)
💡 Detalhes que economizam tempo: st.set_page_config precisa ser a primeira chamada st.* do arquivo, senão dá erro. E use width="stretch" — o antigo use_container_width=True está deprecated.
3. Testar local
streamlit run app.py   # abre em http://localhost:8501
4. Deploy no Streamlit Community Cloud (grátis)
  1. Commit + push do app.py e do requirements.txt atualizado.
  2. share.streamlit.io → login com GitHub → New app.
  3. Selecionar: repositório, branch main, main file app.py.
  4. Advanced settings → Secrets — formato TOML (é diferente do .env! tem aspas e espaços): 
    SUPABASE_URL = "postgresql://postgres.xxx:senha@aws-1-sa-east-1.pooler.supabase.com:5432/postgres"
  5. Deploy. O log de build aparece na própria tela.
🐞 Mesma história do IPv6 aqui: o Streamlit Cloud também não tem saída IPv6. Session pooler, sempre.
Critério de pronto: URL pública do Streamlit mostrando os KPIs vindos do Gold.

Fechar o ciclo e documentar

Provar que as pontas se conectam sem nenhum passo manual no meio.

~30 min
1. O teste de ponta a ponta
  1. Adicione uma linha nova no Excel de movimentação (ex.: loja L03, um SKU novo, 20 unidades ENTRADA) — e o SKU correspondente no Excel de produtos, senão a categoria sai como SEM CATEGORIA.
  2. git add . && git commit -m "..." && git push
  3. Aba Actions → Run workflow → esperar ficar verde.
  4. Conferir no Supabase que a linha nova chegou em dw_gold.posicao_estoque.
  5. Abrir o Streamlit e ver a mudança refletida.
🐞 Armadilha real (você vai cair de novo): "rodei tudo, o Supabase mudou, mas o Streamlit continua igual". Não é bug do pipeline — é o @st.cache_data(ttl=600) servindo o resultado de até 10 minutos atrás. Espere o TTL ou use o menu ⋮ → Clear cache e Rerun.

Método de diagnóstico (vale pra qualquer pipeline): antes de culpar o ETL, consulte o banco direto. Se o dado está lá, o problema é do frontend (cache), não do backend.
2. Escrever o README.md

São três lugares independentes, cada um com seu próprio mecanismo. Trocar a senha do banco significa trocar nos três.

3. Anotar as limitações — isso vira a entrada do Nível 2
  • Fonte é Excel manual — sem contrato de schema; mudar uma coluna quebra o extract().
  • Sem testes de dados (dbt tests, Great Expectations) — os bugs de qualidade foram pegos no olho, não por um teste que falha no CI.
  • load() é replace/truncate total — sem carga incremental, sem histórico, sem trilha de auditoria.
  • Orquestração é um step linear — o Actions não modela dependência entre tarefas, retries granulares nem paralelismo. É o argumento concreto pra existir o Airflow/Dagster.
  • Sem alertas de falha — se o Actions quebrar, ninguém é avisado.
Critério de pronto (Nível 1 concluído): Excel → Actions → Supabase → Streamlit funcionando de ponta a ponta, na nuvem, com README.

2 · Cola rápida: as armadilhas

Se algo quebrar daqui a 6 meses, comece por aqui — a chance de ser um destes é alta.

SintomaCausaCorreção
Network is unreachable com um IP tipo 2600:... Connection string em modo Direct connection → IPv6. Nuvem não tem IPv6. Trocar para Session pooler no Supabase, nos 3 lugares (.env, secret do Actions, secret do Streamlit).
Could not parse SQLAlchemy URL Secret com lixo: prefixo SUPABASE_URL=, aspas ou espaço. Recadastrar só a URI pura.
Datas erradas (dia virou mês) format="mixed", dayfirst=True corrompe datas já em ISO. Separar ISO × BR por str.contains("-"), parsear cada um, concat().sort_index().
Lojas duplicadas no Gold (L02 ≠ L002) Normalização de string incompleta. str.extract(r"(\d+)") + "L" + zfill(3).
requirements.txt ilegível/estranho > do PowerShell grava em UTF-16LE. pip freeze | Out-File -Encoding utf8 requirements.txt.
Streamlit não mostra o dado novo @st.cache_data(ttl=600) servindo cache velho. Esperar 10 min ou ⋮ → Clear cache + Rerun. Confirme antes que o dado está no Supabase.
Deploy do Streamlit falha no build requirements.txt sem streamlit (esqueceu de regenerar). Regenerar, commitar, push — o deploy refaz sozinho.
git commit recusa ("who are you?") Identidade Git não configurada na máquina. git config --global user.name / user.email.
Nome do repo com - no lugar do acento GitHub não aceita acento e substitui em silêncio. Nomear em ASCII desde o início (rename depois obriga a atualizar git remote set-url e reconectar o Streamlit).
Cron não dispara Repo sem push há ~60 dias → GitHub pausa schedule. Qualquer push religa. workflow_dispatch nunca é pausado.

A regra que vale mais que o passo a passo. A ordem das etapas não é estética: cada uma isola uma classe de erro. Rodar o ETL sem banco (1) separa erro de dado de erro de rede. Testar a conexão com o one-liner (2) separa erro de infra de erro de código. Rodar o Actions manualmente (5) separa erro de ambiente de erro de cron. Se você pular etapa, não economiza tempo — só junta várias causas possíveis num erro só, e aí o debug custa o dobro.
