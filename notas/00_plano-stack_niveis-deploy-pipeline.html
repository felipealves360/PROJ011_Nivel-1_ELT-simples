Plano de Stack — Deploy de Pipeline de Dados

Três níveis de evolução, do mais simples ao mais eficiente, para dominar deploy de pipelines na arquitetura Medalhão — de planilhas hoje a extração de ERP e dashboard sobre o Gold.

Planilhas / ERP→ Bronze→ Silver→ Gold→ Dashboard / App
1

Simples — foco em aprender o ciclo completo

Ponta a ponta na nuvem o mais rápido possível, sem se afogar em ferramentas.

🥉 Base

Objetivo: dominar deploy, agendamento e o consumo do Gold com o mínimo de peças móveis.

CamadaTecnologia
ExtraçãoPython + pandas — Excel hoje; requests p/ API do ERP depois
Transformaçãopandas (o que você já tem)
ArmazenamentoSupabase (Postgres) — Bronze / Silver / Gold em schemas
OrquestraçãoGitHub Actions (cron) — sem Airflow/Dagster ainda
DeployRailway ou o próprio GitHub Actions
DashboardStreamlit lendo o schema Gold
Por quê: elimina o orquestrador dedicado e usa um cron do GitHub Actions. Streamlit entrega um dashboard funcional em Python puro em poucas horas. É o caminho mais rápido para ter tudo rodando na nuvem de ponta a ponta.
2

Intermediário (revisado) — dbt + Dagster

Orquestração baseada em assets + transformação versionada e testável.

🥈 Padrão de mercado

Objetivo: orquestração de verdade orientada a ativos de dados, com transformação declarativa, testada e documentada.

CamadaTecnologia
ExtraçãoPython + conectores do ERP (API REST, ou pyodbc/SQLAlchemy se for SQL Server/Oracle)
Ingestão BronzePython → Postgres (raw), encapsulado como asset Dagster
Transformação Silver/Golddbt — cada modelo vira um asset via @dbt_assets
ArmazenamentoSupabase / Postgres
OrquestraçãoDagster (Docker) — schedules + sensors
DeployDocker + Railway (ou VM)
DashboardMetabase ou Streamlit sobre o Gold
Por que Dagster no lugar do Airflow:
  • Assets, não tarefas: você declara bronze → silver → gold como ativos de dados; o lineage aparece sozinho na UI — casa com a arquitetura Medalhão.
  • Integração nativa com dbt: carrega o projeto dbt e transforma cada modelo em asset automaticamente (no Airflow exige plugin/wrapper).
  • DX melhor: roda local com dagster dev (um comando), sem webserver + scheduler + banco de metadados separados.
  • Observabilidade: materializações, logs e checks de qualidade por asset já vêm na interface.
Único custo: comunidade menor que a do Airflow — menos respostas prontas no Google, mas documentação oficial excelente.

3

Eficiente / moderno — padrão de mercado atual

Extração declarativa e orquestração assíncrona que escala.

🥇 Escala

Objetivo: stack que escala, com extração de conectores prontos e destino em warehouse quando crescer.

CamadaTecnologia
Extração / Ingestãodlt (data load tool) ou Airbyte — conectores prontos p/ ERPs e APIs
ArmazenamentoPostgres/Supabase agora; DuckDB local p/ dev; BigQuery se crescer
Transformaçãodbt (Silver / Gold)
OrquestraçãoDagster (asset-based) — schedules + sensors
DeployDocker + Railway/Fly.io, ou Dagster Cloud
Dashboard / AppNext.js + API consumindo o Gold, ou Evidence.dev (BI como código)
Por quê: dlt/Airbyte resolvem o problema real de extrair de ERP sem escrever conector do zero. Dagster + dbt já vêm do Nível 2 — a diferença aqui é a camada de extração (conector manual → dlt/Airbyte) e o destino (Postgres → warehouse). Isso cria um caminho de evolução limpo, sem retrabalho de orquestração.

Recomendação para o seu momento

Como o objetivo é aprender deploy, comece pelo Nível 1 para fechar o ciclo completo (extração → nuvem → dashboard) rápido. Depois evolua para o Nível 2 trocando duas peças de cada vez:

  1. Adicione dbt para Silver/Gold (maior retorno de aprendizado);
  2. Plugue Streamlit ou Metabase no Gold;
  3. Só então troque a fonte Excel pela API do ERP;
  4. Introduza o Dagster quando o pipeline já estiver estável.
⚠️ Erro comum: começar pelo orquestrador. Dagster/Airflow são orquestradores, não o coração do pipeline. O coração é extração confiável + transformação testada (dbt) + Gold bem modelado.
